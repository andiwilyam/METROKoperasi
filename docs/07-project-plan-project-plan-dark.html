1. Pendahuluan

1.1 Tujuan Dokumen

Dokumen Rencana Proyek ini menguraikan rencana pelaksanaan implementasi penuh sistem MetroCoop untuk koperasi klien. Dokumen ini menjadi panduan operasional seluruh tim proyek mulai dari fase Discovery hingga berakhirnya masa Warranty Support.

1.3 Asumsi

  • Klien menyediakan akses penuh ke data legacy untuk keperluan migrasi (jika ada)
  • Klien menunjuk seorang Project Sponsor yang berwenang menyetujui deliverable dan change request
  • Tim klien tersedia untuk sesi workshop, review, dan UAT sesuai jadwal yang disepakati
  • Infrastruktur cloud (Railway + PostgreSQL) disediakan dan dibayar oleh pengembang selama warranty
  • Konektivitas internet stabil tersedia di lokasi klien untuk pelatihan remote
  • Regulasi dan kebijakan internal koperasi tidak berubah selama masa implementasi
  • Data migrasi kurang dari 50.000 record (anggota + transaksi historis)
  • Customisasi UI/UX terbatas pada penyesuaian brand (logo, warna, nama koperasi)
  • Integrasi eksternal (API bank, payment gateway) menggunakan koneksi yang sudah tersedia
  • Durasi hari kerja: Senin–Jumat, 09:00–17:00 WIB

1.5 Risiko Utama

Risiko detail dibahas di Bagian 7: Manajemen Risiko. Ringkasan risiko tinggi:

  • Keterlambatan persetujuan BRD/SRS menggeser seluruh timeline
  • Perubahan scope selama development menambah durasi dan biaya
  • Kualitas data legacy rendah, mempersulit migrasi
  • Keterbatasan pengetahuan teknis tim klien terhadap sistem baru
  • Gangguan infrastruktur cloud selama fase kritis

7.2 Risk Response Strategy

StrategiPenerapan
AvoidHindari risiko dengan menghilangkan penyebab — misalnya freeze scope setelah SRS approval

TransferPindahkan risiko ke pihak lain — misalnya asuransi infrastruktur cloud, SLA dari provider
AcceptTerima risiko dengan contingency plan — misalnya buffer timeline untuk minor delays

10.3 Definition of Done (DoD)

Sebuah user story dianggap done ketika:

  • ✅ Code ditulis dan berfungsi sesuai spesifikasi
  • ✅ TypeScript compilation tanpa error (tsc --noEmit pass)
  • ✅ Build production sukses (vite build + esbuild pass)
  • ✅ Unit test ditulis dan pass (coverage > 80% untuk modul baru)
  • ✅ Integration test pass untuk API endpoints terkait
  • ✅ E2E test pass untuk user flow utama (jika applicable)
  • ✅ Code review disetujui minimal 1 reviewer
  • ✅ Tidak ada security vulnerability baru (npm audit clean)
  • ✅ Responsive di desktop (1280px+) dan mobile (375px+)
  • ✅ Accessibility: keyboard navigable, aria labels untuk interactive elements
  • ✅ Error handling untuk semua edge cases
  • ✅ Audit trail tercatat untuk setiap operasi create/update/delete

10.5 Code Quality Standards

  • TypeScript Strict Mode: Seluruh codebase menggunakan strict: true
  • Linting: ESLint dengan konfigurasi proyek, zero warning di production build
  • Formatting: Prettier dengan config .prettierrc
  • Commit Convention: Conventional Commits (feat:, fix:, docs:, refactor:, test:, chore:)
  • Branch Strategy: Trunk-based development dengan feature branches singkat (< 3 hari)
  • Review Requirement: Minimal 1 approval sebelum merge ke main

11.4 Rollback Plan

Jika migrasi gagal atau data tidak valid:

  • Sistem production menggunakan data snapshot sebelum migrasi

  • Sistem lama tetap aktif sebagai fallback selama 7 hari pasca cutover

  • Rollback script tersedia untuk menghapus data migrated dan mengembalikan state

  • Decision to rollback harus diambil dalam 4 jam pasca cutover
